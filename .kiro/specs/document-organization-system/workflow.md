# Document Organization Workflow

このワークフローは、ワークスペース内のドキュメントを自動的に整理し、クリーンアップの推奨事項を生成します。

## ワークフロー概要

このワークフローは、以下のステップで構成されています：

1. **設定の読み込み** - 設定ファイルまたはデフォルト設定を使用
2. **ドキュメントスキャン** - ワークスペース内のドキュメントファイルを再帰的にスキャン
3. **分析** - メタデータ収集と参照関係の分析
4. **評価** - 重要度スコアの計算とステータスの割り当て
5. **レポート生成** - 整理結果レポートの生成
6. **レビュー** - ユーザーによるレポートの確認
7. **確認** - 推奨アクションの実行確認
8. **アーカイブ** - バックアップ用アーカイブディレクトリの作成（オプション）
9. **クリーンアップ** - ユーザーが実行するコマンドの提供

## ワークフローステップ詳細

### ステップ 1: 設定の読み込み

**目的**: システムの動作を制御する設定を読み込みます。

**アクション**:
- `config.json`ファイルが存在する場合は読み込み
- ファイルが存在しない場合はデフォルト設定を使用
- 設定の妥当性を検証

**設定項目**:
- スキャン対象パス（includePaths）
- 除外パス（excludePaths）
- 対象ファイル拡張子（fileExtensions）
- 評価の重み付け（weights）
- ステータス判定の閾値（thresholds）
- 出力設定（outputPath, includeMetadata）

**出力**: 検証済みの設定オブジェクト

**エラーハンドリング**:
- 無効な設定の場合、エラーメッセージを表示してデフォルト設定を使用
- 設定ファイルが読み取れない場合、警告を表示してデフォルト設定を使用

---

### ステップ 2: ドキュメントスキャン

**目的**: ワークスペース内のすべての対象ドキュメントファイルを検出します。

**アクション**:
- 設定で指定されたディレクトリを再帰的にスキャン
- 除外パターンに一致するディレクトリをスキップ
- 対象拡張子（.md, .txt）のファイルを収集
- 各ファイルの相対パスを記録

**進捗表示**:
```
スキャン中: .agent/
スキャン中: .kiro/specs/
スキャン中: backlog/
...
発見されたドキュメント: 127 ファイル
```

**出力**: スキャンされたドキュメントのリスト

**エラーハンドリング**:
- アクセスできないディレクトリはスキップし、警告をログに記録
- 読み取りエラーが発生したファイルはスキップし、エラーリストに追加

---

### ステップ 3: メタデータ収集と参照関係分析

**目的**: 各ドキュメントの詳細情報を収集し、ドキュメント間の関係を分析します。

**アクション**:

#### 3.1 メタデータ収集
- ファイル作成日時を取得
- 最終更新日時を取得
- ファイルサイズ（バイト）を計算
- 行数をカウント
- ファイルパスからカテゴリを判定（spec, workflow, skill, backlog, docs, config, other）

#### 3.2 参照関係分析
- ドキュメント内容をスキャンしてマークダウンリンクを抽出
- ファイルパス参照を抽出
- 参照グラフを構築
- 各ドキュメントの入力参照数（他のドキュメントから参照されている数）をカウント

#### 3.3 重複チェック
- 各ドキュメントのコンテンツハッシュを計算
- 完全一致の重複を検出
- ファイルサイズと行数に基づく類似ドキュメントを検出

**進捗表示**:
```
メタデータ収集中: 127/127 ファイル
参照関係分析中...
  - 発見された参照: 342 個
  - 参照グラフ構築完了
重複チェック中...
  - 完全一致の重複: 3 組
  - 類似ドキュメント: 7 組
```

**出力**: 
- メタデータ付きドキュメントリスト
- 参照グラフ
- 重複情報マップ

**エラーハンドリング**:
- メタデータ取得に失敗した場合、デフォルト値を使用し、ドキュメントを「要確認」としてマーク
- 参照抽出エラーは警告としてログに記録し、処理を継続

---

### ステップ 4: 重要度評価とステータス割り当て

**目的**: 各ドキュメントの重要度を評価し、適切なステータスを割り当てます。

**アクション**:

#### 4.1 重要度スコア計算
各ドキュメントに対して、以下の要素を考慮してスコア（0-100）を計算：

- **参照数スコア（重み: 0.4）**: 他のドキュメントから参照されている数
  - 参照が多いほど高スコア
  
- **最新性スコア（重み: 0.3）**: 最終更新日からの経過時間
  - 最近更新されたファイルほど高スコア
  
- **サイズスコア（重み: 0.1）**: ファイルサイズと行数
  - 適度なサイズのファイルに高スコア
  
- **カテゴリスコア（重み: 0.2）**: ドキュメントのカテゴリ
  - spec, workflow, skill: 高スコア
  - backlog, docs: 中スコア
  - other: 低スコア

- **重複ペナルティ**: 重複ドキュメントはスコアを減点

#### 4.2 ステータス割り当て
スコアに基づいてステータスを決定：

- **必要（スコア ≥ 60）**: 重要なドキュメント、保持推奨
- **不要（スコア ≤ 30 かつ参照なし）**: 削除候補
- **要確認（30 < スコア < 60）**: ユーザーの判断が必要

**特別ルール**:
- 重複ドキュメントの場合、1つを除いてすべて「不要」に設定

**進捗表示**:
```
重要度評価中: 127/127 ファイル
ステータス割り当て完了:
  - 必要: 45 ファイル
  - 不要: 32 ファイル
  - 要確認: 50 ファイル
```

**出力**: ステータス付きドキュメントリスト

---

### ステップ 5: レポート生成

**目的**: 整理結果を包括的なマークダウンレポートとして生成します。

**アクション**:
- サマリーセクションの生成（統計情報）
- ステータス別にドキュメントをグループ化
- 各ドキュメントの詳細情報を含める：
  - ファイルパス
  - メタデータ（サイズ、更新日、カテゴリ）
  - 重要度スコアと内訳
  - 判断理由
  - 推奨アクション
- 削除推奨セクションの生成（コマンド例付き）
- エラーサマリーセクションの生成（エラーがある場合）

**レポート構造**:
- サマリー（統計情報）
- 必要なドキュメント（詳細リスト）
- 不要なドキュメント（詳細リスト）
- 要確認のドキュメント（詳細リスト）
- 削除推奨（コマンド例）
- エラーサマリー（エラーがある場合）

**出力**: 
- マークダウン形式のレポート文字列
- ワークスペースルートに保存されたレポートファイル

**エラーハンドリング**:
- レポートファイルの書き込みに失敗した場合、エラーメッセージを返す

---

### ステップ 6: レビュー（ユーザー確認ポイント）

**目的**: 生成されたレポートをユーザーに提示し、確認を求めます。

**アクション**:
- レポートファイルのパスを表示
- レポートのサマリーを表示
- ユーザーにレポートの確認を促す

**ユーザーへのメッセージ**:
```
✓ ドキュメント整理レポートが生成されました

レポートファイル: DOCUMENT_ORGANIZATION_REPORT.md

サマリー:
  - 総ドキュメント数: 127
  - 必要: 45 ファイル
  - 不要: 32 ファイル
  - 要確認: 50 ファイル

レポートを確認してください。
次のステップに進む前に、推奨事項を確認することをお勧めします。
```

**ユーザー確認ポイント**:
- [ ] レポートの内容を確認しましたか？
- [ ] 「不要」とマークされたドキュメントのリストを確認しましたか？
- [ ] 削除前にバックアップを作成しますか？

---

### ステップ 7: 確認（ユーザー確認ポイント）

**目的**: 推奨アクションの実行についてユーザーの明示的な確認を取得します。

**アクション**:
- ユーザーに次のアクションについて確認を求める
- 選択肢を提示

**ユーザーへの質問**:
```
推奨されたアクションを実行しますか？

オプション:
1. はい - アーカイブを作成してクリーンアップコマンドを表示
2. いいえ - ワークフローを終了（レポートは保存されています）
3. カスタム - 特定のファイルのみ処理

選択してください: 
```

**ユーザー確認ポイント**:
- [ ] 推奨されたアクションを実行する準備ができていますか？
- [ ] 重要なファイルが誤って削除されないことを確認しましたか？

**分岐**:
- 「はい」→ ステップ 8 へ
- 「いいえ」→ ワークフロー終了
- 「カスタム」→ ユーザーとの対話的な処理

---

### ステップ 8: アーカイブ（オプション）

**目的**: 削除前にドキュメントのバックアップを作成します。

**アクション**:
- アーカイブディレクトリを作成（例: `.archive/documents-YYYYMMDD-HHMMSS/`）
- 「不要」とマークされたドキュメントをアーカイブにコピー
- アーカイブの場所を記録

**コマンド例**:
```bash
# アーカイブディレクトリの作成
mkdir -p .archive/documents-20240115-143022

# 不要なドキュメントをアーカイブにコピー
cp path/to/unnecessary/doc1.md .archive/documents-20240115-143022/
cp path/to/unnecessary/doc2.md .archive/documents-20240115-143022/
```

**出力**:
```
✓ アーカイブが作成されました: .archive/documents-20240115-143022/
  - アーカイブされたファイル: 32 個
  - 総サイズ: 456 KB
```

**注意**: このステップは自動実行されません。ユーザーが手動でコマンドを実行する必要があります。

---

### ステップ 9: クリーンアップ

**目的**: ユーザーが実行できるクリーンアップコマンドを提供します。

**アクション**:
- 削除コマンドのリストを生成
- 移動コマンドのリストを生成（オプション）
- コマンドをファイルに保存（例: `cleanup_commands.sh`）

**生成されるコマンド例**:
```bash
#!/bin/bash
# ドキュメントクリーンアップコマンド
# 生成日時: 2024-01-15 14:30:22
# 
# 警告: これらのコマンドを実行する前に、必ずバックアップを作成してください！
# 
# 推奨: まずアーカイブを作成してから実行してください

# 不要なドキュメントの削除
rm "backlog/old-feature-idea.md"  # 理由: 参照なし、2年以上更新なし
rm "docs/deprecated-guide.md"     # 理由: 重複ドキュメント
rm ".agent/temp-notes.md"         # 理由: 参照なし、一時ファイル

echo "クリーンアップ完了"
```

**出力**:
```
✓ クリーンアップコマンドが生成されました: cleanup_commands.sh

次のステップ:
1. cleanup_commands.sh の内容を確認
2. 必要に応じてコマンドを編集
3. 実行: bash cleanup_commands.sh

または、個別にファイルを削除することもできます。
```

**安全性に関する注意**:
- **このワークフローは自動的にファイルを削除しません**
- すべての削除はユーザーが手動で実行する必要があります
- 削除前に必ずバックアップを作成することを推奨します

## ワークフロー実行モード

### インタラクティブモード（デフォルト）

ユーザーとの対話を含む完全なワークフローを実行します。

**実行方法**:
```bash
# Kiro Skillとして実行
kiro run document-organization-system

# または、直接実行
npm run organize
```

**特徴**:
- 各ステップで進捗を表示
- ユーザー確認ポイントで一時停止
- 対話的な選択肢を提供

---

### 非インタラクティブモード

ユーザー確認なしでレポート生成まで自動実行します。

**実行方法**:
```bash
# 非インタラクティブフラグ付きで実行
kiro run document-organization-system --non-interactive

# または
npm run organize -- --non-interactive
```

**特徴**:
- レポート生成まで自動実行
- ユーザー確認をスキップ
- ファイルの削除は行わない（レポートのみ生成）
- CI/CDパイプラインでの使用に適している

---

### ドライランモード

実際の変更を行わずにワークフローをシミュレートします。

**実行方法**:
```bash
# ドライランフラグ付きで実行
kiro run document-organization-system --dry-run

# または
npm run organize -- --dry-run
```

**特徴**:
- すべてのステップをシミュレート
- ファイルシステムへの変更なし
- 生成されるコマンドのプレビュー
- テストや検証に適している

## エラーハンドリング

### エラー発生時の動作

ワークフロー実行中にエラーが発生した場合：

1. **回復可能なエラー**（ファイル読み取り失敗など）:
   - エラーをログに記録
   - 処理を継続
   - エラーサマリーをレポートに含める

2. **致命的なエラー**（設定が無効、出力ディレクトリに書き込めないなど）:
   - エラーメッセージを表示
   - ワークフローを中断
   - 部分的な結果を保存（可能な場合）

### エラーリカバリー

```
エラーが発生しました: ファイル 'path/to/file.md' を読み取れません

オプション:
1. スキップして続行
2. リトライ
3. ワークフローを中断

選択してください: 
```

## ベストプラクティス

### 実行前の準備

1. **バックアップの作成**: 重要なファイルのバックアップを作成
2. **設定の確認**: `config.json` で適切なパスと閾値を設定
3. **テスト実行**: まず `--dry-run` モードで実行して結果を確認

### レポート確認時のチェックポイント

1. **「不要」ドキュメントの確認**: 本当に削除して問題ないか確認
2. **「要確認」ドキュメントの判断**: 手動で必要性を判断
3. **重複ドキュメントの確認**: どちらを保持するか確認

### 安全な削除手順

1. アーカイブを作成
2. 少数のファイルから開始
3. 段階的に削除を実行
4. 各段階で動作を確認

## トラブルシューティング

### よくある問題

**問題**: スキャンが遅い
- **原因**: 大量のファイルまたは大きなディレクトリ
- **解決策**: `excludePaths` を設定して不要なディレクトリを除外

**問題**: 重要なファイルが「不要」とマークされる
- **原因**: 評価基準の閾値が適切でない
- **解決策**: `config.json` の `thresholds` を調整

**問題**: 参照が正しく検出されない
- **原因**: 非標準的なリンク形式
- **解決策**: 参照抽出ロジックをカスタマイズ（将来の拡張機能）

## まとめ

このワークフローは、ドキュメント整理プロセスを自動化し、安全で透明性の高い方法でワークスペースをクリーンアップするのに役立ちます。

**重要な原則**:
- **安全第一**: 自動削除は行わず、ユーザーの確認を必須とする
- **透明性**: すべての判断理由を明確に記録
- **柔軟性**: 設定可能な評価基準と閾値
- **監査可能**: すべてのアクションをログに記録

**次のステップ**:
1. `config.json` を作成してカスタマイズ
2. ドライランモードで実行してテスト
3. レポートを確認して評価基準を調整
4. 本番実行でワークスペースを整理
